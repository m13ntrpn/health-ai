# План: несколько пользователей бота

## Текущее состояние

**Бэкенд уже поддерживает нескольких пользователей:**

- Процедуры для бота (`getDailyLogForTelegramUser`, `listDailyLogsForTelegramUser`, `upsertDailyLogForTelegramUser`) принимают **`telegramUserId`** в каждом запросе.
- `ensureUserByTelegramId(telegramUserId)` находит или создаёт пользователя по Telegram id; данные (дневники, приёмы пищи и т.д.) привязаны к `userId`.
- Разные люди = разные `telegramUserId` → разные пользователи в БД → изоляция данных.

**Что должно быть на стороне бота/OpenClaw:**

- При каждом вызове нашего API передавать **`telegramUserId`** того, кто написал в Telegram (например, `message.from.id` из update).
- Один общий `X-Service-Token` для доступа к API; пользователь определяется только через `telegramUserId` в теле запроса.

---

## Что нужно сделать

### 1. Интеграция OpenClaw / Fatherbot (вне репозитория)

- Убедиться, что в запрос к нашему API попадает **Telegram user id** того, кто отправил сообщение.
- В Telegram API это обычно `message.from.id` (число). Мы принимаем строку, можно передать `String(message.from.id)`.
- Контракт: при добавлении записи в дневник вызывать, например,  
  `healthLog.upsertDailyLogForTelegramUser({ telegramUserId: "<id>", date: "YYYY-MM-DD", payload: { ... } })`.

Никаких изменений в коде nadin-health для этого не требуется — только правильная подстановка `telegramUserId` на стороне бота.

---

### 2. Документация в репозитории (рекомендуется)

- В README или в отдельном файле описать:
  - что бот может использоваться несколькими пользователями;
  - что при каждом вызове API нужно передавать `telegramUserId` из Telegram (например, `message.from.id`);
  - пример тела запроса для одной из процедур.
- Так проще подключать и менять интеграцию с OpenClaw/Fatherbot.

---

### 3. Опционально: процедура «текущий пользователь» для бота

- Добавить процедуру вида `service.getMe` или `user.getByTelegramId`: вход — `telegramUserId` (при наличии `X-Service-Token`), выход — минимальная информация о пользователе (id, можно позже — имя из профиля).
- Нужна, если бот должен показывать что-то вроде «вы залогинены как …» или подставлять имя в ответы. Не обязательна для мультипользовательского режима как такового.

---

### 4. Веб-интерфейс (отдельная задача)

- Сейчас веб (Logs, Dashboard) показывает данные одного пользователя из `DEFAULT_TELEGRAM_USER_ID`.
- Чтобы **несколько людей** могли заходить на сайт и видеть каждый свои данные, нужна отдельная авторизация (например, [Telegram Login Widget](https://core.telegram.org/widgets/login)), сохранение сессии по Telegram id и подстановка этого id в запросы вместо `DEFAULT_TELEGRAM_USER_ID`. Это не входит в минимальный план «ботом могут пользоваться больше чем 1 человек».
